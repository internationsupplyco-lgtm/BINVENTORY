Account recovery

Reset login code

Client-side app: recovery needs the workspace on this device or Firestore, plus a recovery key set earlier.
  1. Enter workspace code, username, and recovery key
  2. Own key resets that account; Core’s key can reset any non-core account
  3. Set a new login code (recovery key stays the same)
Use that account’s key, or Core’s key to reset a non-core user.

Need the whole workspace from cloud backup? Use Recover workspace instead (workspace recovery key).

Workspace recovery

Restore workspace

Uses the workspace recovery key (not a personal account key). Loads auto-saved backup from Firestore when sync is on, or an invite/snapshot if you paste one.
  1. Enter workspace code + workspace recovery key
  2. Optional: paste invite/snapshot if cloud isn’t available
  3. After verify: reattach locally and optionally reset Core’s login code

Leave blank to keep Core’s current login code. Workspace stays attached either way.

Forgot only a user’s login code? Use Account recovery on the sign-in screen.

      Not configured

      Firebase web config for project inventory-23fb1 is already embedded in this app (GitHub Pages visitors get cloud sync without pasting). Override below only if you rotate keys. Edits auto-save to Firestore (workspaces/{CODE}) after a short debounce — status shows Saving / Saved / Backup failed.

      1. Firebase Console → Authentication → Get started → enable Anonymous
      2. Create Firestore → Rules → paste/publish firestore.rules from this repo
      3. Core: First-time setup → share ?ws=YOURCODE with friends
      4. Optional: paste a different config → Save & enable (runs a live read/write test)

      Connected means Anonymous Auth + a successful Firestore probe (not just that config JSON loaded).

      Pre-filled from embedded FIREBASE_CONFIG. Save stores an override in localStorage on this device. Only password digests are written to Firestore — never plaintext.
      Firestore rules notes (demo)

      Warning: open rules are for personal demos only. After editing firestore.rules / storage.rules, republish in Firebase Console. Ticket screenshots use Storage paths workspaces/{CODE}/tickets/{ticketId}/shot.jpg (preferred) or legacy tickets/{CODE}/{ticketId}.jpg.

      rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // Require Anonymous Auth + valid workspace code (publish firestore.rules).
    // Also covers connection probe doc workspaces/_SYNC_PROBE.
    match /workspaces/{workspaceCode} {
      allow read, write: if request.auth != null
        && workspaceCode.matches('^[A-Z0-9_\\-]{2,48}$');
      match /tickets/{ticketId} {
        allow read, write: if request.auth != null
          && workspaceCode.matches('^[A-Z0-9_\\-]{2,48}$');
      }
      match /appeals/{appealId} {
        allow read, write: if request.auth != null
          && workspaceCode.matches('^[A-Z0-9_\\-]{2,48}$');
      }
      match /versions/{versionId} {
        allow read, write: if request.auth != null
          && workspaceCode.matches('^[A-Z0-9_\\-]{2,48}$');
      }
    }
  }
}
